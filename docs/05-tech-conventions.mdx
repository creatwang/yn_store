## 1. 项目结构约定

### 1.1 Monorepo 组织

```
my-ecommerce/
├── apps/                    # 应用层
│   ├── admin/               # Next.js 15 管理后台
│   ├── storefront/          # Astro 5 店铺前端

│   ├── db/                  # Drizzle ORM schema & 查询
│   ├── services/            # 共享业务逻辑

│   └── shared/              # 共享类型、工具函数

├── turbo.json               # Turborepo 任务配置
├── pnpm-workspace.yaml      # pnpm 工作空间配置
└── package.json             # 根 package.json
```

### 1.2 命名规范

| 类型 | 规范 | 示例 |
|------|------|------|
| 文件名 | `kebab-case` | `product-list.tsx`, `use-cart.ts` |
| React 组件 | `PascalCase` | `ProductList`, `CartDrawer` |
| 函数 / 变量 | `camelCase` | `getProducts`, `cartTotal` |
| 常量 | `UPPER_SNAKE_CASE` | `API_BASE_URL`, `DEFAULT_LIMIT` |
| 类型 / 接口 | `PascalCase` | `Product`, `OrderResponse` |
| 数据库表名 | `snake_case` | `product`, `order_item` |
| 数据库列名 | `snake_case` | `created_at`, `deleted_at` |
| CSS 类名 | `kebab-case` (Tailwind 优先) | `text-primary`, `btn-outline` |

### 1.3 包结构

每个 `packages/*` 必须包含：

```
packages/db/
├── src/
│   ├── index.ts             # 统一导出
│   ├── schema/              # 表定义
│   └── queries/             # 查询函数
├── package.json             # 包名: @ecommerce/db
├── tsconfig.json            # 继承根 tsconfig
└── README.md                # 包说明
```

包命名统一使用 `@ecommerce/` 作为 scope：

- `@ecommerce/db`
- `@ecommerce/services`
- `@ecommerce/api-client`
- `@ecommerce/shared`

### 2.2 表定义规范

```typescript
import { pgTable, text, timestamp, jsonb, numeric, boolean } from "drizzle-orm/pg-core";

export const product = pgTable("product", {
  id: text("id").primaryKey(),
  title: text("title").notNull(),
  subtitle: text("subtitle"),
  description: text("description"),
  handle: text("handle").unique(),
  is_giftcard: boolean("is_giftcard").default(false).notNull(),
  status: text("status").default("draft").notNull(),
  thumbnail: text("thumbnail"),
  metadata: jsonb("metadata"),
  created_at: timestamp("created_at", { withTimezone: true }).defaultNow().notNull(),
  updated_at: timestamp("updated_at", { withTimezone: true }).defaultNow().notNull(),
  deleted_at: timestamp("deleted_at", { withTimezone: true }),
});
```

关键规则：

- **表名**与 Medusa 现有表名保持完全一致（`snake_case`），不要加前缀或后缀
- **id 字段**统一使用 `text` 类型（Medusa 使用字符串 ID）
- **时间戳**统一使用 `timestamp("xxx", { withTimezone: true })`
- **软删除**：所有表都有 `deleted_at` 字段

### 2.3 BigNumber 字段处理

Medusa 使用 BigNumber 存储金额，实际表中存两列：

```typescript
export const lineItem = pgTable("cart_line_item", {
  // ... 其他字段

  unit_price: numeric("unit_price"),
  raw_unit_price: jsonb("raw_unit_price"),

  quantity: numeric("quantity"),
  raw_quantity: jsonb("raw_quantity"),
});
```

- `amount` 列（`numeric`）：用于数据库排序和聚合
- `raw_amount` 列（`jsonb`）：存储精确值 `{ value: "1999", precision: 20 }`，前端展示时使用

### 2.4 软删除查询

所有查询默认需要排除已软删除的记录：

```typescript
import { isNull } from "drizzle-orm";

const products = await db
  .select()
  .from(product)
  .where(isNull(product.deleted_at));
```

建议封装为查询工具函数：

```typescript
export function notDeleted<T extends { deleted_at: unknown }>(table: T) {
  return isNull(table.deleted_at);
}

const products = await db
  .select()
  .from(product)
  .where(notDeleted(product));
```

### 2.5 关系定义

使用 Drizzle `relations()` 定义模块内关系：

```typescript
import { relations } from "drizzle-orm";

export const productRelations = relations(product, ({ many }) => ({
  variants: many(productVariant),
  options: many(productOption),
  images: many(productImage),
}));

export const productVariantRelations = relations(productVariant, ({ one, many }) => ({
  product: one(product, {
    fields: [productVariant.product_id],
    references: [product.id],
  }),
  prices: many(price),
}));
```

### 3.4 表单处理

使用 **React Hook Form** + **Zod** 处理所有表单：

```typescript
import { useForm } from "react-hook-form";
import { zodResolver } from "@hookform/resolvers/zod";
import { z } from "zod";

const productSchema = z.object({
  title: z.string().min(1, "商品名称不能为空"),
  description: z.string().optional(),
  status: z.enum(["draft", "published", "rejected"]),
  handle: z.string().regex(/^[a-z0-9-]+$/, "只允许小写字母、数字和连字符"),
});

type ProductFormValues = z.infer<typeof productSchema>;

function ProductForm() {
  const form = useForm<ProductFormValues>({
    resolver: zodResolver(productSchema),
    defaultValues: { title: "", status: "draft" },
  });

  return (
    <Form {...form}>
      <form onSubmit={form.handleSubmit(onSubmit)}>
        {/* 表单字段 */}
      </form>
    </Form>
  );
}
```

### 3.5 数据获取

使用 **TanStack Query v5** 管理所有服务端状态：

```typescript
import { useQuery, useMutation, useQueryClient } from "@tanstack/react-query";

export function useProducts(params: ListProductsParams) {
  return useQuery({
    queryKey: ["products", params],
    queryFn: () => adminClient.products.list(params),
  });
}

export function useCreateProduct() {
  const queryClient = useQueryClient();

  return useMutation({
    mutationFn: (data: CreateProductInput) => adminClient.products.create(data),
    onSuccess: () => {
      queryClient.invalidateQueries({ queryKey: ["products"] });
    },
  });
}
```

Query Key 规范：

- `["products"]` — 商品列表
- `["products", { offset, limit, status }]` — 带参数的商品列表
- `["products", id]` — 单个商品
- `["products", id, "variants"]` — 商品的变体

### 3.7 API 代理

通过 Next.js API Routes 代理 Medusa Admin API：

```typescript
// src/app/api/admin/[...path]/route.ts
import { NextRequest, NextResponse } from "next/server";

const MEDUSA_BACKEND_URL = process.env.MEDUSA_BACKEND_URL || "http://localhost:9000";

async function handler(req: NextRequest, { params }: { params: Promise<{ path: string[] }> }) {
  const { path } = await params;
  const targetUrl = `${MEDUSA_BACKEND_URL}/admin/${path.join("/")}${req.nextUrl.search}`;

  const response = await fetch(targetUrl, {
    method: req.method,
    headers: {
      "Content-Type": "application/json",
      Authorization: req.headers.get("Authorization") || "",
    },
    body: req.method !== "GET" ? await req.text() : undefined,
  });

  const data = await response.json();
  return NextResponse.json(data, { status: response.status });
}

export { handler as GET, handler as POST, handler as DELETE };
```

### 4.4 React Islands

交互组件使用 React Islands 模式：

| 指令 | 用途 | 使用场景 |
|------|------|---------|
| `client:load` | 页面加载时立即 hydrate | 购物车、结账表单、导航菜单 |
| `client:visible` | 滚入视口时 hydrate | 商品评价、推荐商品 |
| `client:idle` | 浏览器空闲时 hydrate | 非关键交互组件 |
| `client:only="react"` | 仅客户端渲染 | 依赖 window/document 的组件 |

## 6. API 设计规范

> 以下规范适用于自建 API Routes（Next.js API Routes 或独立 API 服务）。调用 Medusa API 时遵循 Medusa 自身的规范。

### 6.1 RESTful 风格

| 操作 | 方法 | 路径 | 示例 |
|------|------|------|------|
| 列表 | `GET` | `/api/admin/{resource}` | `GET /api/admin/products` |
| 详情 | `GET` | `/api/admin/{resource}/{id}` | `GET /api/admin/products/prod_01` |
| 创建 | `POST` | `/api/admin/{resource}` | `POST /api/admin/products` |
| 更新 | `POST` | `/api/admin/{resource}/{id}` | `POST /api/admin/products/prod_01` |
| 删除 | `DELETE` | `/api/admin/{resource}/{id}` | `DELETE /api/admin/products/prod_01` |

### 6.2 列表响应格式

```json
{
  "products": [
    { "id": "prod_01", "title": "T恤" }
  ],
  "count": 100,
  "offset": 0,
  "limit": 20
}
```

### 6.3 单条响应格式

```json
{
  "product": {
    "id": "prod_01",
    "title": "T恤",
    "variants": []
  }
}
```

### 6.4 错误响应格式

```json
{
  "type": "not_found",
  "message": "Product with id prod_01 not found"
}
```

常见错误类型：

| type | HTTP Status | 说明 |
|------|------------|------|
| `invalid_data` | 400 | 请求数据验证失败 |
| `unauthorized` | 401 | 未认证 |
| `not_allowed` | 403 | 无权限 |
| `not_found` | 404 | 资源不存在 |
| `duplicate_error` | 409 | 资源冲突/重复 |
| `unexpected_state` | 409 | 资源状态不允许该操作 |
| `unknown_error` | 500 | 服务器内部错误 |

### 6.5 分页与排序

- **分页参数**：`offset`（偏移量，默认 0）+ `limit`（每页数量，默认 20，最大 100）
- **排序参数**：`order=created_at:desc` 或 `order=title:asc`
- **多字段排序**：`order=status:asc,created_at:desc`

### 6.6 字段选择与关系展开

- **字段选择**：`fields=id,title,status` — 只返回指定字段
- **关系展开**：`expand=variants,variants.prices` — 展开嵌套关系

### 6.7 认证

所有 Admin API 需要携带 JWT：

```
Authorization: Bearer eyJhbGciOiJIUzI1NiIs...
```

## 7. Git 规范

### 7.1 分支策略

```
main           ← 生产环境，只通过 PR 合并
  └── develop  ← 开发主分支
       ├── feature/admin-product-list    ← 功能分支
       ├── feature/store-checkout        ← 功能分支
       ├── fix/cart-calculation          ← 修复分支
       └── refactor/db-schema            ← 重构分支
```

分支命名规范：

- `feature/xxx` — 新功能
- `fix/xxx` — Bug 修复
- `refactor/xxx` — 代码重构
- `docs/xxx` — 文档更新
- `chore/xxx` — 构建/工具更新

### 7.2 Commit 规范

使用 [Conventional Commits](https://www.conventionalcommits.org/) 规范：

```
<type>(<scope>): <subject>

<body>

<footer>
```

类型：

| type | 说明 | 示例 |
|------|------|------|
| `feat` | 新功能 | `feat(admin): 添加商品列表页面` |
| `fix` | Bug 修复 | `fix(cart): 修复数量计算错误` |
| `docs` | 文档更新 | `docs: 更新架构文档` |
| `refactor` | 代码重构 | `refactor(db): 重构 product schema` |
| `style` | 样式调整 | `style(admin): 调整侧边栏布局` |
| `test` | 测试 | `test(services): 添加订单服务测试` |
| `chore` | 构建/工具 | `chore: 更新 turborepo 配置` |
| `perf` | 性能优化 | `perf(store): 优化商品列表加载` |

### 7.3 Pull Request 规范

每个 PR 必须包含：

1. **标题**：遵循 Conventional Commits 格式
2. **描述**：
   - 变更内容概述
   - 相关 Issue 链接
   - 截图（UI 变更时）
3. **测试计划**：
   - 手动测试步骤
   - 自动化测试覆盖情况
4. **Review Checklist**：
   - [ ] TypeScript 类型完整
   - [ ] 软删除查询已处理
   - [ ] 错误处理完善
   - [ ] 国际化文案已添加
   - [ ] 响应式布局已测试
