### 三层回滚体系

#### 三层回滚体系

```
Workflow 引擎             → DB ─→ 外部服务 ─→ compensate ─→ 下一个 step
                          8 个跨服务 + 外部调用的核心流程

dispatchRollbackProcess  → try { action() } catch { rollback() }
                          非 workflow 但有业务回滚逻辑的场景

db.transaction()         → PostgreSQL 原生回滚
                          纯 DB 跨表操作，无外部调用
```

#### Workflow 引擎升级计划

| 当前 | 升级后 |
|------|--------|
| handler 签名 `({ input })` | `({ input, output, providers })` — 可读前序 step 输出 + 注入 provider |
| compensate 拿不到 step 数据 | compensate 可访问 step 已产生的数据，精确回滚 |
| createWorkflow 纯 step 数组 | 新增 `providers` 参数，自动注入 |

#### `dispatchRollbackProcess` — 独立于引擎的业务回滚

```ts
// lib/rollback.ts
export async function dispatchRollbackProcess<T>(
  action: () => Promise<T>,
  rollback: (result: T | null, error: Error) => Promise<void>,
): Promise<T>
```

适用于不需要多步编排、但需要"失败时做业务回滚"的场景（如 order-edit addItems、claim addItems 等）。与 Workflow 引擎不冲突——引擎管多步编排，`dispatchRollbackProcess` 管单步回滚。

详见 **[docs/workflow-plan.md](workflow-plan.md)** §4。

#### 不用 Workflow 也不需 dispatchRollbackProcess 的场景

| 场景 | 原因 |
|------|------|
| order.cancel / complete | 纯 DB 操作 |
| fulfillment.cancel | 纯 DB 操作 |
| fulfillment.delivered | DB + 通知。通知 fire-and-forget 无需补偿 |
| order-edit confirm | 纯 DB 操作（apply actions），价格差手动处理 |
| 所有通知/邮件 | fire-and-forget，失败不影响业务数据 |

## C-02: fields 字段按需加载

### 状态 🟡 部分完成（基础设施就绪，部分路由未接线）

### 已实现：Orders 模块

`services/order/fields.ts` 提供完整 fields 解析：
- `parseOrderFields(fieldsParam)` — 逗号分隔 → string[]
- `resolveOrderFieldsConfig(fields)` — 解析 wantsFulfillments / wantsCustomer / wantsSummary 等标记
- `DEFAULT_ADMIN_ORDER_RETRIEVE_FIELDS` — 30+ 字段对齐官方 `defaultAdminOrderFields`
- `DEFAULT_ADMIN_ORDER_LIST_FIELDS` — 列表默认字段集
- `applyOrderFieldMask(dto, fieldConfig)` — 按 fields 裁剪响应

### 调用链验证

```
GET /admin/orders     → orderService.list(query)
  query.fields 来自 listOrdersSchema（Zod validator 包含 fields?: string）
  → presentAdminOrders(db, orders, { fields: query.fields })
  → resolveOrderFieldsConfig(fields)
  ✅ 列表 fields 全链路通

GET /admin/orders/:id  → orderService.getById(id)
  实际调用：orderService.getById(c.req.param("id"))  ← 未传 fields!
  → presentAdminOrderDetail(db, item, DEFAULT_ADMIN_ORDER_RETRIEVE_FIELDS)
  ❌ 详情 fields 未从 query string 接线（固定用默认集）
```

### 仍缺

| 端点 | 问题 |
|------|------|
| `GET /admin/orders/:id` | 路由 handler 未读 `c.req.query("fields")`，未传给 `getById()` |
| `GET /admin/products/:id` | 产品详情同样未接线 fields query |
| `GET /admin/products` | 产品列表 fields 存在但未确认全链路 |
| Store 端订单/产品 | 无 fields 支持，返回固定结构 |

### 候选方案

- **短平快**：在 `GET /admin/orders/:id` handler 里补 `const fields = c.req.query("fields")`，透传给 `getById(id, false, undefined, fields)`
- **完整方案**：在 products 和 store 模块复用同一 `parseFields` + field mask 模式

## C-03: 订单地址管理

### 状态 ✅ 已解决（2026-06-02）

### 调用链验证

**读链路** — `presentAdminOrderDetail()` 通过 FK 加载地址：
```typescript
// services/order/admin-order.ts:974-979
orderRow.shipping_address_id
  ? db.select().from(orderAddress).where(eq(orderAddress.id, orderRow.shipping_address_id))
  : Promise.resolve(null)
```
响应体包含 `shipping_address` 和 `billing_address`。

**写链路** — 三处全部补完：

| 写入点 | 做了什么 |
|--------|---------|
| `orderService.create()` | 接收 shipping_address / billing_address → INSERT INTO order_address → 写入 FK |
| `orderService.update()` | 同上，支持 null（清除地址） |
| `cartService.completeCheckout()` | 从 cart_address 迁移到 order_address，设置 FK |

**Validator 对齐** — `packages/validators/src/order.ts`：
- `addressSchema` 对齐 Medusa `AddressPayload`（字段一致 + `nullish()` + `.strict()` + metadata）
- `createOrderSchema` + `updateOrderSchema` 均包含 `shipping_address` / `billing_address`

### 解决方式

拷贝 Medusa 官方 `common-validators/common.js` 的 `AddressPayload` 定义（字段、nullish、strict），适配到项目 Zod schema。写入逻辑新建 address 行并盖 FK。

### 已知局限

- `cartService` 的 `getById()` 不返回 cart addresses，`completeCheckout` 直接查 `cart_address` 表而非通过 service 封装
- `orderService.update()` 每次更新地址会新建 `order_address` 行，旧行不变（Medusa 同为新建行——地址不可变原则）

## C-06: 通知系统

### 状态 🟡 Module 抽象层就绪（2026-06-02），多渠道/模板引擎/notification 表直接读写仍可改进

### 已实现

**notificationService**（`services/notification.service.ts`）— 对齐 Medusa `notificationModuleService`：

```typescript
// 调用方
notificationService.send({
  to: customer.email,
  template: "order.confirmed",
  data: { display_id, order_id },
  trigger_type: "order.placed",
  resource_id: orderId,
  resource_type: "order",
  idempotency_key: `order-confirm-${orderId}`,
```

**接入点**（6 处，替换原 fire-and-forget 裸调用）：

**邮件函数** — 8 个保持不变（`lib/mail.ts`），但调用链路从 `service → mail()` 改为 `service → notificationService.send(..., sender: () => mail())`。

**事件订阅者** — `event-subscribers.ts` 不再直接发邮件，改为 console.log（邮件在 service 层通过 notificationService 触发）。订阅者职责收敛为审计/日志/未来扩展点。
