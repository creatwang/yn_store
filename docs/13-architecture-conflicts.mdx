# 架构冲突记录 — Medusa 官方 vs Hono 自定义实现

> 本文档记录因架构差异导致无法直接对齐 Medusa 官方实现的点。
> 每个冲突列出现状、官方做法、影响范围、候选方案，逐项讨论决策。

## C-01: Workflow 编排层

### 状态 ✅ 已解决

### 现状
已实现轻量 Workflow 引擎（`apps/server/src/lib/workflow.ts`），支持 Step 定义 + 补偿回调。
两个核心 Workflow 已迁移：

| Workflow | 步骤数 | 文件 |
|----------|--------|------|
| create-fulfillment | 8 steps | `workflows/create-fulfillment.ts` |
| cancel-fulfillment | 6 steps | `workflows/cancel-fulfillment.ts` |

Workflow 引擎支持：
- 顺序执行步骤
- 步骤失败时逆序执行补偿
- 步骤间通过 `StepContext` 共享数据

### 官方差异（有意为之）
- 官方用函数闭包传递数据（`createWorkflow(name, fn)`），我们用数组定义 + `StepContext`
- 官方支持 `StepResponse`/`StepError` 返回值，我们直接返回对象
- 上层效果等价

### 待迁移
- create-shipment
- create-return / receive-return
- cancel-order / complete-order
- 索赔/换货

## C-02: fields 字段选择系统

### 现状
API 返回固定字段集，客户端无法控制返回哪些字段/关联。

### 官方做法
Medusa 通过 `req.queryConfig.fields` + `refetchEntity` 实现按需加载：
```ts
// query-config.ts 定义默认字段
defaultAdminRetrieveOrderFields = [
  "*items.*tax_lines",
  "*shipping_address",
  "*payment_collections.*payments",
]

// route.ts 按 fields 参数加载
const product = await refetchEntity({
  entity: "product",
  idOrFilter: req.params.id,
  fields: selectFields,  // 只查这些
})
```

`*` 前缀表示展开关联对象，无 `*` 只返回外键。

### 影响范围
- GET /admin/products/:id（已部分实现 fields 解析）
- GET /admin/orders/:id（未实现 fields 解析）
- GET /admin/orders（列表未支持 fields）

### 候选方案
- **A**: 为 orders 和 products 分别实现 fields 解析器（类似官方的 `wants()` 函数）
- **B**: 保持当前固定返回，后续按需扩展
- **C**: 实现通用 `parseFields()` 工具函数，各 route 复用

## C-03: 地址管理

### 现状
`order` 表无 `shipping_address_id` / `billing_address_id` 字段。
地址数据存在 `metadata` 中，非结构化。

### 官方做法
Medusa 使用 `order_address` 表存储地址，order 通过 `shipping_address_id` / `billing_address_id` 引用。
`AdminUpdateOrder` 可以直接传 `shipping_address` / `billing_address` 完整对象。

```sql
order_address (id, customer_id, company, first_name, ...)
order (id, ..., shipping_address_id FK, billing_address_id FK)
```

### 影响范围
- POST /admin/orders/:id（更新地址）
- POST /admin/orders（创建时带地址）
- GET /admin/orders/:id（响应含 `*shipping_address` 和 `*billing_address`）

### 候选方案
- **A**: 给 `order` 表加 `shipping_address_id` / `billing_address_id` 列，地址写入 `order_address`
- **B**: 保持 metadata 存储，地址作为 JSON 写入
- **C**: 在 `order_address` 表存地址记录，但通过 metadata 引用 ID

## C-04: Order Totals 计算

### 现状
`order_summary` 表存储 JSON 格式的 totals，但未做结构化计算。
API 响应不包含 `subtotal`、`tax_total`、`discount_total` 等顶层字段。

### 官方做法
Medusa 在 OrderSummary Module 中自动计算 totals：
```ts
// 计算逻辑（简化）
item_total = Σ (item.unit_price × item.quantity) - discounts
shipping_total = Σ shipping_methods.amount
tax_total = Σ tax_lines.rate × taxable_amount
total = item_total + shipping_total + tax_total
```

### 影响范围
- GET /admin/orders/:id 响应缺少 `total`, `subtotal`, `tax_total`, `discount_total`, `item_total`, `shipping_total`
- Admin 订单详情页依赖这些字段显示金额

### 候选方案
- **A**: 在 `getById` 中加 Drizzle SQL 计算 totals
- **B**: 在 order_service 中加 `computeTotals()` 函数
- **C**: 接受现状，前端自己计算（不推荐）

## C-05: Versioning / Order Change Actions

### 现状
`orderChange` 表记录了变更元信息，但无 `order_change_action` 表。
所有操作（添加商品、修改配送）不会拆分为独立的 Action 记录。

### 官方做法
Medusa 的每个订单编辑操作都创建 `order_change_action` 记录：
```ts
// 添加商品 → 创建 action
action = { id, type: "ITEM_ADD", reference: "product_variant", reference_id, amount, ... }
// 修改数量 → 创建 action
action = { id, type: "ITEM_UPDATE", reference: "order_item", reference_id, new_quantity, ... }
// 确认编辑 → 批量应用所有 pending actions
```

### 影响范围
- 订单编辑工作流（/admin/order-edits）
- 索赔/换货编辑（inbound/outbound items）
- 草稿订单编辑
- GET /admin/orders/:id/changes（返回 actions 列表）

### 候选方案
- **A**: 新建 `order_change_action` 表，实现 action 追踪
- **B**: 将 actions 作为 JSON 存在 `orderChange.metadata` 中
- **C**: 保持现状，actions 隐式写在其他表中

## C-06: 通知系统

### 现状
无通知模块。所有 API 的 `no_notification` 参数存在但被忽略。

### 官方做法
Medusa Notification Module 处理邮件/SMS/webhook 通知：
```ts
await notificationService.create({
  to: customer.email,
  channel: "email",
  template: "order.confirmed",
  data: { order_id, ... }
})
```

### 影响范围
- 订单确认通知
- 发货通知
- 退货确认

### 候选方案
- **A**: 接入第三方通知服务（如 Resend/SendGrid）
- **B**: 暂时忽略 no_notification，后续再实现
- **C**: 加一个简单的 console.log 通知桩

## C-07: 事件系统

### 现状
无事件总线。操作完成后不触发任何事件。

### 官方做法
Medusa Workflow 在每个 Step 完成后 emit 事件：
```
order.placed, order.canceled, fulfillment.created, return.requested, ...
```

其他模块（Search, Notification, Webhook）通过订阅事件做出响应。

### 影响范围
- 跨模块联动
- 第三方集成
- 审计日志

### 候选方案
- **A**: 引入 EventEmitter / 消息队列
- **B**: 不实现，需要时手动调用
- **C**: 用 Hono middleware 做简单的 after-hook
